    Does this negate the need for a permalink on the front-end, in so much that it contains both the slug and the permalink in the general url?

    Is there a way to be more user friendly? Could there be a bit of error checking that goes on? The vast majority will get that url right; they link straight to it and may never see the link's bits and pieces. All is well and good.

    When you are pulling up the object by its row ID, you can check the object's slug against the url's slug to ensure that this is the article they're looking for. Maybe a notice that their url is a bit off and to use the correct one?

    Or what about they type in the row ID number but get it wrong? What if the row doesn't exist? Could you then use the slug to pull up the article anyway, thus circumventing the 404 page?

    And finally, what about if they forget to add the row ID? Best to safe-guard and have pulling up slug as a fail-safe. It won't happen often and therefore I don't think it will hurt performance that much. Besides a little (minuscule) performance hit for a better user experience is optimal, is it not?

  • bennylope

    I'm not sure I understand your first question, as a the article link as mentioned in the post is a permalink - whether using the prescribed link style or traditional slug-only style.

    Adding a failsafe link would be simple enough. You'd simply add another URL pattern after the above mentioned that looks only for the slug. You should route that to a view which looks up the requested article and redirects (301, permanent!) to the proper URL.

    Beyond that, I'm not convinced there's a great need to do a whole lot in the way of validating the slug. The Onion's site rewrites the slug of the article if it doesn't match the ID. But there's a floor for what we should realistically need to do for every user scenario, at least with regard to things like typos in the slug. If someone tries the URL "/contac/" should the site check to see if it's similar to an existing URL? Maybe, but this isn't expected behavior. The stuff people copy in links they send in email and post to social networking sites (like sending you a link to the last page of a long magazine article) leads me to believe that most people copy links whole without taking the effort to scrutinize every part.

    Lastly, "What if the row doesn't exist? Could you then use the slug to pull up the article anyway, thus circumventing the 404 page?"

    No. If the article doesn't exist, it doesn't matter if you use the ID or the slug, you're not going to be able to retrieve it.
