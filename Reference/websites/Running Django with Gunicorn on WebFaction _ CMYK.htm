Running Django with Gunicorn on WebFaction

If you're like me, you've probably been hearing a lot lately about asynchronous/event-driven/non-blocking web applications and the advantage these have in terms of performance and scalability. You've probably heard of this in the context of Node.js, Tornado, or Comet in general. These technologies are indeed opening the door to whole new classes of web applications and have enormous impact on the degree of interactivity possible on the web. These new applications are a paradigm shift from classical web applications and require fundamentally different implementations; this is a topic for another day. However, from an architecture point-of-view, it's possible for conventional applications to take advantage of async web servers for signficant performance gains with minimal changes to the application code.

This morning I went through the process of transitioning CMYK – a Django site formerly served by Apache with mod_wsgi – to run on the Gunicorn WSGI server with Gevent workers. In this post I'll share some notes on getting this deployed on WebFaction – my awesome Django hosting provider – as well as some performance metrics captured using Blitz.io.

Spoiler: the difference is uncanny.

The Setup

Before I start, I have to give credit to the following resources which helped me along the way:

  • http://dbinit.com/blog/going-green/
  • http://ericholscher.com/blog/2011/jan/10/handling-django-settings-files/
  • http://devcenter.heroku.com/articles/django

My approach is a hodge-podge of the above ideas. Some of the changes I made are not directly related to Gunicorn and are just general improvements I made to the deployment model, specifically making better use of virtualenv and pip and taming my Django settings files, but I'll try to cover everything I did as best as I can remember.

Set up Virtualenv

In your local dev environment, create and activate a new virtualenv, and copy the Django app into it.

cd ~/mysite
mkdir myapp_gunicorn
cd myapp_gunicorn
virtualenv --distribute .
source bin/activate
cp -R ~/mysite/myapp .

Install django, gunicorn, gevent, and whatever else your app needs. (This make take a few tries to get everything.)

pip install django gunicorn gevent
pip install other-shit-i-need

Once you've got everything and your app starts properly (using manage.py for now), save the requirements to a text file which we can use later on WebFaction to mirror the setup.

pip freeze > requirements.txt

The settings file interlude

In the past I'd struggled to find a satisfactory approach to managing Django settings files across different environments. At first, I didn't keep these in the repo at all. For awhile I used a semi-clever approach split out environment specific settings into separate files and import them into the main settings.py file, but annoyingly this required the shell game of specifying which environment-specific module to import. Here's a better way.

Create a new settings directory in the app, organized as follows:

myapp/
    settings/
        __init__.py
        base.py
        dev.py
        prod.py

base.py will contain the general application settings (e.g., INSTALLED_APPS). dev.py and prod.py contain all the environment-specific stuff (e.g., database connections, template paths, debug options, etc.). Rather than importing the environment-specific stuff into the general settings, we do the reverse. At the top of dev.py and prod.py:

from .base import *

Once we do this, we're done using manage.py and lose some magic, but in the end this is a good thing. (Eric Holscher's post linked above still uses manage.py in development, but I prefer my approach which reduces the stack differences to a minimum.)

Now, we'll set up some configuration files for gunicorn in the following structure:

etc/
    gunicorn.dev.conf
    gunicorn.prod.conf

These conf files should look like this:

bind = "127.0.0.1:8000"
workers = 3
worker_class = "gevent"

The prod configuration will differ in the bind port number and possibly the number of workers. FYI, Gunicorn recommends the formula (2 x $num_cores) + 1 as a starting point.

As the coup de grâce, we can use a startup script (start_myapp.sh) to fire up our app:

#!/bin/bash
source bin/activate
export DJANGO_SETTINGS_MODULE=cmyk.settings.$1
gunicorn_django -c etc/gunicorn.$1.conf &

Now we can start our app with the command ./start_myapp.sh $env and the appropriate settings for both Gunicorn and Django will kick in.

You'll probably have some minor debugging to do, particularly being more specific in imports since manage.py is not there to make assumptions for you, but after that your app should be running locally on Gunicorn. (Shut it down with a kill $pid to the Gunicorn master process.)

Deployment in WebFaction

In the Applications section of the WebFaction control panel, create a new Custom app (listening on port) (myapp_gunicorn). Copy the created app's port number into your WebFaction-specific Gunicorn config (etc/gunicorn.prod.conf).

Commit your code, push it up to your repo host of choice, and clone it into your new ~/webapps/myapp_gunicorn directory. If you're using Git, the following .gitignore should leave what you need:

bin
build
include
lib
.Python
*.pyc

Now install virtualenv (if it's not already there), get the dependencies installed, and fire up your app:

cd ~/webapps/myapp_gunicorn
easy_install-2.7 install virtualenv
~/bin/virtualenv --distribute .
source bin/activate
pip install -r requirements.txt
./start_myapp.sh prod

To complete the cutover, go to the Websites section of the WebFaction control panel, and change the appropriate site app from your old Apache-powered Django instance to your new myapp_gunicorn. Once you click Update you should be off to the races. After you've verified your app is working as expected, you can tear down the Apache Django app.

The Bake-Off

I used Blitz.io to measure the difference in performance. Blitz is a handy and massively entertaining load-testing tool which I use as an add-on to CloudFlare, a value-added CDN which I also highly recommend.

It's important to note that, through their partnership, CloudFlare lets the Blitz traffic pass through to the origin servers, so the metrics below are direct hits into the WebFaction Gunicorn server. Therefore, real-world site performance should be even better when the CDN and/or HTTP caching is in play.

I configured the Blitz "rush" as follows:

-p 1-250:60 -T 5000 http://cmyk.sevennineteen.com/

This means: rush the CMYK homepage, increasing traffic from 1 to 250 concurrent users over the course of 1 minute, with a 5 second timeout.

And now, the numbers you've all been waiting for. Again, both versions are running the same Django application code and differ only in the WSGI container.

Apache + mod_wsgi

  • 137 hits, 1,039 timeouts, 25 errors
  • The average hit rate of 2.31/second translates to about 199,772 hits/day
  • The first error happened at 17.59 seconds into the test when the number of concurrent users was at 74

Gunicorn + gevent

  • 2,029 hits, 154 timeouts, 154 errors
  • The average hit rate of 31/second translates to about 2,713,363 hits/day
  • The first timeout happened at 45.20 seconds into the test when the number of concurrent users was at 189

As you can see, the Gunicorn setup is capable of handling over 13.5x more traffic than Apache (and yes, I ran the test several times). I expected there to be some improvement in performance due to the async Gevent workers, but nothing like this.

It's worth noting that 99% of Django sites, including mine, will probably be fine with the basic Apache + mod_wsgi setup, and no doubt the heavy-hitter Django sites that need this kind of scalability (e.g., Instagram, Disqus, Pinterest, The Onion) have already discovered the virtues of Gunicorn/Gevent/Twisted/Tornado. But this is food for thought for fledgling sites starting to tackle scalability concerns, as asynchronous servers can provide a huge amount of headroom with minimal application refactoring.
