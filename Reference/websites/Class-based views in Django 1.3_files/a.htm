    Your `compute_context()` method already exisits, it's called `get_context_data()`. It's also worth noting that you don't need to pass `request` and `**kwargs` everywhere since they're available as class attributes: you can safely use `self.request`, `self.args` and `self.kwargs` (args and kwargs are captured from the URL pattern) from any method of the class that's called after `dispatch()`.

    As for decorating the views I'm a fan of calling `as_view` in views.py like

    views.py:
    my_view = login_required(MyViewClass.as_view())

    Then your urls.py will look the same as with function based views

    urls.py:
    ...
    url(r'^/$', 'my_view', name='myview'),
    ...

  • Diederik van der Boor

    Instead of having to write your own `def get()` everywhere, consider using the `DetailView`.This class already provides the retreival of objects, context, and rendering. For example:

    class ArticleDetailView(DetailView):
        model = Article
        template_name = "article/details.html"

    Et, voila!


    To override the behavior, see the methods:
    - get_context_data()
    - get_object()
    - get_template()

    This way, writing view code almost becomes declarative.

  • Donald Stufft

    Found an error:

    You can't just pass any old kwarg in via as_view. It will throw an exception if the kwarg passed to as_view doesn't already exist as an attribute of the class. Further more anything passed to as_view as a kwarg techincally get's passed as kwargs to the CBV's __init__ method, which the default __init__ method just loops over all the kwargs it is passed and setattr's them onto self.

    So the final result is that passing a kwarg to as_view that isn't already an attribute of the CBV is an exception, and passing a kwarg to as_view that is an attribute will make that kwarg end up available at getattr(self, "kwarg_name")
